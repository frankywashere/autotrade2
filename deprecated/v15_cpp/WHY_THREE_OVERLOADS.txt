================================================================================
                "WHY DO WE HAVE 3 VERSIONS OF extract_all_features()?"
                               EXECUTIVE SUMMARY
================================================================================

Quick Answer:
─────────────
We have 3 overloads because:

  1. COMPATIBILITY: Pass 3 uses vectors, Pass 1/2 uses DataView
  2. COMPLETENESS: Pass 1/2 has channel history, Pass 3 doesn't
  3. LEGACY CODE: Overload #2 is deprecated delegation layer

Reality:
────────
  • We NEED 2 overloads (vector-based and DataView+history)
  • We HAVE 3 overloads (the middle one is unnecessary)
  • Recommendation: Remove Overload #2

================================================================================
                        WHY THESE SPECIFIC DESIGNS?
================================================================================

Question: Why not just use ONE overload with optional history?
Answer:   Different callers have different data types:

  Pass 1/2: Has DataView (zero-copy, efficient)
  Pass 3:   Has vector<OHLCV> (already sliced data)

  Forcing vector → DataView or DataView → vector would add unnecessary copies.


Question: Why does Pass 3 not have history?
Answer:   Architectural limitation:

  Pass 3 processes individual channels in isolation (sliced data from channel
  start to channel end). It cannot see "previous channels" because it only
  has the current channel's data slice.

  Pass 1/2 has full dataset, so it can track channel history as it scans.


Question: Why 670 history features = 0.0 instead of removing them?
Answer:   Feature vector consistency:

  ML model expects EXACTLY 14,840 features in fixed order.

  Missing features would break model predictions.

  Zero-filling maintains vector shape while indicating "no history available".

================================================================================
                              CONCLUSION
================================================================================

"Why do we have 3 versions?"

  Historical Reasons:
    • Started with simple vector-based version
    • Added DataView for memory efficiency
    • Added history features later
    • Created delegation layer for compatibility

  Technical Reasons:
    • Different callers use different data types (vector vs DataView)
    • Different callers have different data availability (history vs no history)

  Reality:
    • We only NEED 2 overloads
    • Overload #2 is legacy delegation that can be removed
    • No caller currently uses Overload #2

  Recommendation:
    • Remove Overload #2
    • Keep Overload #1 (Pass 3, vector-based)
    • Keep Overload #3 (Pass 1/2, DataView + history)


The 3-overload design was a transitional state during feature development.
The code has evolved past the need for the middle delegation layer.
